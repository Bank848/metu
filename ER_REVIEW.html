ER DIAGRAM REVIEW & FIXES

เทียบ crow's-foot diagram (v1) กับ Prisma schema จริง แล้วจด bugs, missing constraints, และการแก้ไขที่ใช้จริงใน production. ใช้เป็นเอกสารประกอบตอน defend โปรเจกต์กับอาจารย์ได้เลย.

4 bugs ผิดชัด 4 PK/Cardinality 5 UNIQUE หาย 1 entity หาย 3 timestamps หาย 3 nullable FK
1

Bugs ใน diagram ที่ผิดชัด

ผิดกฎของ ER / ประเภทข้อมูล / สะกดผิด — ต้องแก้ก่อนนำเสนอ.

Entity.Field ปัญหา แก้เป็น
USER_STATS.buyer_level
USER_STATS.seller_level		 มาร์ค FK ทั้งคู่ แต่เป็นเลขเลเวลเฉย ๆ — ไม่ได้อ้างอิงตารางไหน เอา FK ออก ให้เป็น int ธรรมดา (default 1 / 0)
TRANSACTION.transaction_type มาร์ค FK แต่ไม่มีตาราง TRANSACTION_TYPE อยู่จริง เปลี่ยนเป็น Enum {purchase, payout, refund}
COUPON_USAGE.user_id type string — ไม่ match USER.user_id (int) → JOIN ไม่ได้ เปลี่ยนเป็น int
PRODUCT_ITEM.quntity สะกดผิด quantity
2

Cardinality / PK ที่ไม่ถูกหลัก

PK เลือกผิด หรือไม่สะท้อนความสัมพันธ์ที่ต้องการ.

Entity ปัญหา วิธีแก้
PRODUCT_IMAGE ใช้ product_id เป็น PK → สินค้า 1 ชิ้นมีรูปได้แค่ใบเดียว (demo ต้องการ 3-5 ใบ) เพิ่ม surrogate PK product_image_id + field sort_order int เพื่อจัดเรียงรูป
STORE_STATS มี stat_id PK แยก — อาจเกิด stats หลายแถวต่อ store ทำ UNIQUE(store_id) หรือใช้ store_id เป็น PK ตรง ๆ (1:1 กับ STORE)
STORE.owner_id ไม่มี UNIQUE → user 1 คนเปิดได้หลายร้าน (ขัด business rule) ใส่ UNIQUE(owner_id)
ORDER.cart_id ไม่มี UNIQUE → cart 1 ใบอาจกลายเป็น order หลายใบ ใส่ UNIQUE(cart_id)

Missing UNIQUE constraints

ข้อจำกัดที่ต้องเขียนใน diagram เพื่อกันข้อมูลซ้ำ.

Entity Columns เหตุผล
USER username · email log-in ซ้ำไม่ได้
PRODUCT_REVIEW (product_id, user_id) รีวิวสินค้าเดียวกันได้แค่ครั้งเดียวต่อคน

PRODUCT_TAG tag_name ชื่อ tag ซ้ำไม่ได้ใน catalogue
4

Entity ที่หายไป — CART_ITEM

Diagram v1 เชื่อม CART ↔ ORDER ตรง ๆ แต่ขาดตารางเก็บของในตะกร้า "ก่อน" checkout.

เพิ่มตาราง
CART_ITEM
CART_ITEM
├─ cart_item_id      PK  int
├─ cart_id           FK  → CART
├─ product_item_id   FK  → PRODUCT_ITEM
├─ quantity              int
└─ created_at            timestamp

UNIQUE (cart_id, product_item_id)
Relationship ใหม่
CART ↔ CART_ITEM ↔ PRODUCT_ITEM
  • • CART —crow's-foot— CART_ITEM (1:N)
  • • CART_ITEM —crow's-foot— PRODUCT_ITEM (N:1)
  • • ทำให้ตะกร้าหนึ่งใบเก็บหลายรายการได้ (ไม่งั้นต้องสร้าง cart ใหม่ทุกครั้งที่เพิ่มของ)
  • • ตอน checkout: CART.status → checked_out, สร้าง ORDER + copy cart_items → ORDER_ITEM

Columns ที่ขาด

Timestamps สำหรับ sort / audit และ type ที่ต้อง correct.

Entity ต้องเพิ่ม / แก้ เหตุผล
PRODUCT + created_at timestamp
+ updated_at timestamp		 sort by "newest", audit trail, ดูว่าใครแก้ล่าสุดเมื่อไหร่
PRODUCT_REVIEW + created_at timestamp แสดง "รีวิว 2 วันที่แล้ว", sort ตามล่าสุด
CART.session_id type int → varchar (หรือลบทิ้ง) session token เป็น string ไม่ใช่ int

Nullable FK ที่ไม่ได้มาร์ค

ใน crow's-foot ต้อง วงกลมกลวง ◯ ฝั่ง FK เพื่อบอกว่าเป็น NULL-able.

FK ทำไมต้อง nullable
ORDER.transaction_id Order ที่ status=pending ยังไม่มี transaction (ยังไม่จ่าย)
ORDER_ITEM.coupon_id ไม่ใช่ทุก line ใช้ coupon
USER.country_id user อาจไม่ระบุประเทศตอนสมัคร

Checklist — ลำดับการแก้

ทำตามนี้แล้ว diagram จะตรงกับ schema จริงใน Prisma.

  1. 1. แก้ bugs 4 ข้อในตาราง Bugs (เอา FK ผิดออก, แก้ type, แก้ typo quntity)
  2. 2. แก้ PRODUCT_IMAGE ให้มี surrogate PK product_image_id + sort_order
  3. 3. เพิ่มตาราง CART_ITEM + โยง CART → CART_ITEM → PRODUCT_ITEM
  4. 4. ใส่ UNIQUE บน username, email, owner_id, cart_id, (product_id, user_id), (product_id, tag_id), (store_id, code)
  5. 5. มาร์ค nullable ◯ บน 3 FK: transaction_id, coupon_id, country_id
  6. 6. เพิ่ม timestamps บน PRODUCT (created_at, updated_at) และ PRODUCT_REVIEW (created_at)
  7. 7. เพิ่ม box ข้าง diagram เขียน Enum values ของทั้ง 7 enum ให้ครบ
  8. 8. เขียน Cascade rules ในหมายเหตุใต้ diagram
✓
